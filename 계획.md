## 1. 개요

### 1.1 고찰 작성의 중요성
발굴조사보고서의 고찰(考察) 섹션은:
- 조사 성과를 학술적으로 종합·분석하는 핵심 부분
- 국가유산청 규정상 필수 항목 (68개 체크리스트 준수 필요)
- 작성 기간: 통상 **2주~1개월** 소요
- 자동화 시: **10~15분**으로 단축 가능

### 1.2 파이프라인 구조
```
사전 준비 → STEP 1~6 순차 실행 → 산출물 검토 → 최종 활용
   (1일)        (약 12분)              (1~2일)        (보고서 반영)
```

### 1.3 기대 효과
- ⏱️ **시간 절약**: 수주 작업 → 10분 자동화
- 📊 **품질 보증**: 국가유산청 규정 68개 항목 자동 검증
- 🎓 **학술 엄밀성**: 주변 유적 10개 이상 체계적 비교, 문헌 50개 이상 자동 인용
- 🔄 **재사용 가능**: 다른 유적 조사에도 즉시 적용

## 2. 사전 준비 단계 (D-1 ~ D-Day)

### 2.1 자료 수집 체크리스트

#### ✅ 폴더 구조 생성
```bash
mkdir -p 논문 발굴조사보고서 주변유적 output
```

#### ✅ 필수 자료 수집
| 폴더명 | 내용 | 최소 수량 | 권장 수량 | 파일 형식 |
|--------|------|----------|----------|----------|
| `논문/` | 관련 학술 논문 | 5개 | 10~15개 | PDF |
| `발굴조사보고서/` | 주변 지역 보고서 | 10개 | 20~30개 | PDF, HWP |
| `주변유적/` | 주변 유적 관련 자료 | 5개 | 10개 이상 | PDF, HWP, TXT |

**자료 수집 기준**:
- **공간 범위**: 대상지 중심 반경 20~50km 이내
- **시대 범위**: 대상지 시대 ±1개 시대 (예: 청동기시대 → 신석기~원삼국 포함)
- **주제 관련성**: 유구·유물 유형, 취락 구조, 편년 연구 관련

#### ✅ 조사 정보 정리
아래 정보를 사전에 준비하여 텍스트 파일(`조사정보.txt`)로 저장:

```yaml
# 기본 정보
조사명: "○○지구 유적 발굴조사"
조사기관: "○○문화재연구원"
조사기간: "2024.03.01 ~ 2024.11.30"
조사면적: "5,000㎡"

# 위치 정보
대상지_좌표:
  위도: 37.5665
  경도: 126.9780
  주소: "○○도 ○○시 ○○동 123번지"

# 비교 범위
비교_반경: 20  # km (유적 부족 시 자동 확대)

# 시대 정보
주요_시대:
  - "청동기시대"
  - "원삼국시대"

# 유구 정보
주요_유구:
  - 주거지: 12기 (원형 5기, 방형 7기)
  - 수혈유구: 34기
  - 구상유구: 2기

# 유물 정보
주요_유물:
  - "무문토기 (구연부 26점, 동체부 85점)"
  - "석기 (석촉 12점, 석도 5점, 마제석검 1점)"
  - "적색마연토기 (완형 3점, 파편 12점)"

# 특기사항
특기사항: "환상 배치 구조 확인, 중앙 광장 직경 약 40m"

# 방사성탄소연대 (있을 경우)
C14_연대:
  - 시료: "H-03 노지 목탄"
    측정값: "2,850±40 BP"
    보정연대: "B.C. 1050~920 (2σ)"
    분석기관: "○○연대측정소"
```

## 3. 파이프라인 실행 계획 (D-Day)

### 3.1 전체 실행 방법 (권장)

**실행 명령**:
```
archaeology-orchestrator 스킬을 사용하여 고찰을 작성해주세요.

조사 정보:
- 조사명: ○○지구 유적
- 조사기관: ○○문화재연구원
- 조사기간: 2024.03~2024.11
- 대상지 좌표: 위도 37.5665, 경도 126.9780
- 시대: 청동기시대, 원삼국시대
- 주요 유구: 주거지 12기(원형 5, 방형 7), 수혈 34기
- 주요 유물: 무문토기, 석기(석촉 12, 석도 5, 마제석검 1), 적색마연토기
- 특기사항: 환상 배치 구조, 중앙 광장 직경 40m
```

또는 간단히:
```
고찰작성
```
(오케스트레이터가 대화형으로 정보를 묻습니다)

**예상 소요 시간**: 약 10~15분

### 3.2 단계별 실행 계획

각 단계를 개별 실행하려는 경우 (디버깅, 재실행 등):

#### STEP 1: 자료 수집 및 정규화 (약 2분)
```
data-normalizer 스킬을 실행해주세요.
```

**진행 과정**:
1. 3개 폴더 스캔
2. PDF/HWP 파일 텍스트 추출
3. 메타데이터 파싱 (제목, 저자, 연도, 유적명, 좌표 등)
4. JSON Lines 형식으로 저장

**출력 확인**:
```bash
ls -lh output/normalized/
# - documents.jsonl      : 전체 문서 내용
# - metadata.csv         : 메타데이터 요약
# - extraction_log.txt   : 처리 로그
```

**체크 포인트**:
- [ ] 총 파일 수와 처리 성공 수 확인 (예: 42개 중 40개 성공)
- [ ] metadata.csv 열람하여 유적명, 좌표 추출 여부 확인
- [ ] extraction_log.txt에서 오류 메시지 확인

**예상 문제 및 해결**:
| 문제 | 원인 | 해결 방안 |
|------|------|----------|
| HWP 파일 읽기 실패 | 구버전 HWP | 최신 버전으로 재저장 또는 PDF 변환 |
| PDF 텍스트 추출 불가 | 이미지 스캔 PDF | OCR 처리 (Adobe Acrobat 등) |
| 좌표 추출 실패 | 보고서 내 좌표 미기재 | 수동 입력 또는 지명 지오코딩 |

#### STEP 2: 주변 유적 매칭 (약 3분)
```
similarity-matcher 스킬을 실행해주세요.

대상지 정보:
- 위도: 37.5665
- 경도: 126.9780
- 비교 반경: 20km
- 주요 시대: 청동기시대, 원삼국시대
```

**진행 과정**:
1. documents.jsonl에서 좌표 있는 유적 필터링
2. Haversine 공식으로 거리 계산
3. 시대·유구·유물 유사도 계산 (Jaccard Index)
4. 종합 점수로 랭킹, 상위 10개 선정

**출력 확인**:
```bash
cat output/compare/nearby_ranked.csv
# 열: 유적명, 거리(km), 시대_유사도, 유구_유사도, 종합_점수
```

**체크 포인트**:
- [ ] 반경 20km 내 유적이 5개 이상인가?
  - 부족 시: 반경 자동 확대 (20→50→100km)
- [ ] 상위 10개 유적의 종합 점수가 0.3 이상인가?
  - 낮을 시: 시대 범위 확장 또는 자료 추가 수집
- [ ] 주요 비교 유적 5개 이상 확보되었는가?

**예상 문제 및 해결**:
| 문제 | 원인 | 해결 방안 |
|------|------|----------|
| 주변 유적 5개 미만 | 자료 부족 | 1) 비교 반경 확대 (50km, 100km)<br>2) 보고서 추가 수집<br>3) 시대 범위 확장 |
| 좌표 없는 유적 다수 | 메타데이터 추출 실패 | 주요 유적 좌표 수동 입력 |

#### STEP 3: 규정 체크리스트 생성 (약 1분)
```
regulation-checker 스킬을 실행해주세요.
```

**진행 과정**:
1. 국가유산청 규정 68개 항목 로드
2. 현재 수집 자료와 대조
3. 충족/불충족 판정
4. 결손 항목 보고서 생성

**출력 확인**:
```bash

# 68개 항목별 체크 상태

# 불충족 항목 및 보완 방안
```

**체크 포인트**:
- [ ] 충족률 70% 이상인가?
  - 미만 시: gaps.md 확인하여 추가 자료 수집
- [ ] C14 연대, 도면, 사진 등 필수 자료 있는가?

**주요 규정 항목 예시**:
- 조사 개요 (위치, 면적, 기간)
- 유구 종류별 수량 및 특징
- 유물 형식 분류 및 편년
- 주변 유적 비교 (최소 5개)

#### STEP 4: 고찰 초안 작성 (약 5분) ⭐ **핵심**

**진행 과정**:

   - I. 서론 및 조사 성과 종합

- [ ] 전체 분량 30~50쪽인가?
- [ ] 주변 유적 비교 섹션이 5쪽 이상인가?
- [ ] 본문 인용 50개 이상인가?
- [ ] 표 5개 이상, 도면 언급 10개 이상인가?

**품질 자가 점검**:
| 항목 | 기준 | 확인 |
|------|------|------|
| 분량 | 30~50쪽 | ☐ |

```
tables-appendix 스킬을 실행해주세요.
```

**체크 포인트**:
- [ ] 주변 유적 비교표에 10개 유적 포함되었는가?
- [ ] 참고문헌이 본문 인용과 일치하는가?
- [ ] 참고문헌 형식이 통일되었는가? (저자(연도), 『제목』, 출판사)

#### STEP 6: 규정 준수 검증 및 개선 (약 2분)
```
compliance-audit 스킬을 실행해주세요.
```

**진행 과정**:
1. discussion.md를 68개 규정 항목과 대조
2. 정량 기준 검증 (분량, 인용 수, 표 개수 등)
3. 인용 무결성 검사 (본문↔참고문헌 일치)
4. 논리 일관성 검증 (연대 모순, 수량 불일치)
5. 개선안 생성 → discussion_v2.md

**출력 확인**:
```bash
cat output/final/compliance_report.md
# 준수율 확인 (목표: 85% 이상)

cat output/final/change_log.md
# v1 → v2 변경 사항
```

**체크 포인트**:
- [ ] 준수율이 85% 이상인가?
  - 미만 시: compliance_report.md에서 미흡 항목 확인하여 수동 보완
- [ ] 인용 무결성 100%인가? (본문 인용 = 참고문헌 목록)
- [ ] 연대 모순, 수량 불일치 없는가?

**준수율 등급**:
- **우수**: 90% 이상 → 즉시 활용 가능
- **양호**: 80~89% → 미흡 항목 1~2개 수동 보완
- **보통**: 70~79% → 주요 항목 재검토 필요
- **미흡**: 70% 미만 → 자료 추가 수집 후 재실행

### 8.2 자주 묻는 질문 (FAQ)

#### Q1: 자료가 충분하지 않으면 어떻게 하나요?
**A**: 최소 기준 (논문 5개, 보고서 10개)만 충족하면 실행 가능합니다. 다만 준수율이 낮을 수 있으므로, 산출물 검토 후 추가 자료를 수집하여 재실행하는 것을 권장합니다.

#### Q2: HWP 파일을 읽을 수 없어요.
**A**: 구버전 HWP는 호환 문제가 있을 수 있습니다. 최신 버전으로 재저장하거나, PDF로 변환하여 사용하세요.

#### Q3: 좌표 정보가 없는 경우는?
**A**: 1) 지명으로 지오코딩 시도 (geopy 패키지), 2) Google Maps 등에서 수동 확인, 3) 보고서 내 지도에서 추정

#### Q4: 주변 유적이 5개도 안 나와요.
**A**: similarity-matcher가 자동으로 반경을 확대합니다 (20→50→100km). 그래도 부족하면 시대 범위를 넓히거나 (예: 청동기 → 신석기~원삼국), 추가 보고서를 수집하세요.

#### Q5: C14 연대 측정 자료가 없어요.
**A**: 고찰에서 형식학적 편년으로 대체하고, "향후 과제"에서 절대연대 측정 필요성을 언급하면 됩니다.

#### Q6: 참고문헌이 본문 인용과 안 맞아요.
**A**: compliance-audit에서 자동 수정을 시도하지만, 100% 완벽하지 않을 수 있습니다. references.md를 직접 편집하여 누락·오기를 수정하세요.

#### Q7: 준수율이 70% 미만이에요.
**A**: compliance_report.md의 "미흡 항목"을 확인하여:
1. 추가 자료 수집 가능 항목 → 수집 후 재실행
2. 조사 자체에서 없는 항목 (예: 무덤 유구) → "향후 과제"에 언급

#### Q8: 고찰 내용이 우리 유적과 맞지 않아요.
**A**: 조사정보.txt를 더 구체적으로 작성하거나, discussion-drafter 재실행 시 추가 정보를 입력하세요. 또는 discussion_v2.md를 직접 수정하세요.

#### Q9: 파이프라인 실행 중 오류가 났어요.
**A**: output/*/extraction_log.txt 파일에서 오류 메시지를 확인하고, 해당 스킬만 재실행하세요. 대부분 자료 파일 문제(암호화, 손상 등)입니다.

**A**: 각 유적별로 별도 폴더를 만들어 순차 실행하거나, 프로젝트를 복제하여 병렬 실행할 수 있습니다.
